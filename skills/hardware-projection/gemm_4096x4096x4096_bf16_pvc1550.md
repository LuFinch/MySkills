# Performance Projection Report: GEMM 4096×4096×4096 BF16

## Hardware
- **Device**: Intel Data Center GPU Max 1550 (single tile)
- **Architecture**: Xe-HPC (Ponte Vecchio)
- **Hardware Path**: XMX Engine (BF16)
- **Peak Compute (BF16 XMX)**: 419.4 TFLOPS (per tile, official spec: 838.86 TFLOPS full card)
- **Peak Compute (FP32 VE)**: 26.2 TFLOPS (per tile)
- **Memory Bandwidth**: 1600 GB/s (per tile, official spec: 3276.8 GB/s full card, 8× HBM2e stacks @ 3.2 Gbps)
- **Ridge Point (BF16 XMX)**: 262.1 FLOP/Byte
- **EUs**: 512 (per tile), Max Clock: 1600 MHz
- **Source**: `xpu_models/intel_data_center_gpu_max_1550_8086_0bd5.md` + official Intel specs

## Operator: GEMM (MatMul)
- **Shape**: [4096, 4096] × [4096, 4096] → [4096, 4096]
- **Data Type**: BF16 (2 bytes per element)
- **Total FLOPs**: 2 × 4096 × 4096 × 4096 = **137,438,953,472 ≈ 137.44 GFLOP**
- **Total Bytes**: (4096² + 4096² + 4096²) × 2 = 3 × 16,777,216 × 2 = **100,663,296 bytes ≈ 100.66 MB**
  - A (input): 4096 × 4096 × 2 = 33.55 MB
  - B (input): 4096 × 4096 × 2 = 33.55 MB
  - C (output): 4096 × 4096 × 2 = 33.55 MB
- **Arithmetic Intensity**: 137,438,953,472 / 100,663,296 = **1365.3 FLOP/Byte**
- **Bound Type**: **Compute-Bound** (AI 1365.3 >> Ridge Point 262.1)

## Optimization Suggestions

Since this GEMM is **compute-bound**:

1. **Ensure XMX utilization**: The kernel must use DPAS/XMX instructions (not scalar VE). Verify via profiling that XMX pipes are active.
2. **Tile size tuning**: Optimize work-group tile dimensions (e.g., 256×256 or 128×128 tiles) to match XMX systolic array depth and maximize register reuse.
3. **Occupancy**: Balance register pressure vs. occupancy — too many registers per thread reduces concurrent work-groups.
4. **Data layout**: Use column-major or tiled layout for B matrix to maximize cache line utilization and avoid bank conflicts in SLM.
5. **Prefetching**: Use HW or SW prefetch for the next tile while computing the current tile to hide memory latency.
6. **Use oneMKL/oneDNN**: For production, Intel's tuned GEMM libraries typically achieve 70-80%+ efficiency on PVC for large square GEMMs.

## Notes

- This projection is for a **single tile** (512 EUs). For multi-tile execution, data partitioning and inter-tile communication overhead must be considered.
- The 4096³ GEMM is large enough to saturate the XMX engines with good tiling. Smaller sizes (e.g., M or N < 512) may significantly reduce efficiency.
- Actual performance depends heavily on kernel implementation quality, memory allocation (device vs. shared), and runtime overhead.
